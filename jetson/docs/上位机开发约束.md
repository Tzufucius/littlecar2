# 上位机开发约束

## 职责边界

1. `app/` 只负责任务决策与业务编排，通过 `Car` 调用底盘；不得导入 `frame.py` 或 `transport.py` 自行拼帧。
2. `vision/` 只输出识别结果，不持有串口对象，不直接触发电机命令。
3. `protocol/` 是唯一串口所有者。一个物理串口只能创建一个 `Car` / `SerialTransport` 实例。
4. 设备原始协议仅保留在 STM32 的 `drive_*`、`sensor_*` 中；上位机不得透传或依赖其帧格式。

## 安全与时序

1. 正常运动前必须完成 `ping`、启动心跳、清除可恢复安全锁、使能底盘。
2. `HeartbeatRunner` 的周期必须小于 `0.3 s`；默认 `0.1 s`。若 `last_error` 非空，业务层必须停止发送运动命令并进入安全处置。
3. 对 `ProtocolTimeoutError`、串口异常和非 `ACK_OK`，业务层必须立即停止后续运动计划；在链路仍可用时调用 `safe_stop(mode=1)`。
4. 急停、停止、取消目标不等待当前任务步骤完成。新任务下发前应明确是否需要 `cancel_goal()`。
5. `goto_pose()` 不等于到达。只有 `get_motion_status().state == MotionState.ARRIVED` 才能进入下一任务步骤。
6. 所有实车测试默认以零速度、车轮架空或划定安全区进行。非零速度、`goto_pose` 和解除安全锁必须显式由测试人员启用。

## 协议与兼容性

1. 帧格式、CRC、字节序、`CmdSet`、`CmdID` 与 Payload 字段必须以 `MDK-ARM/docs/上下位机通信协议.md` 为准。
2. 新能力优先新增 `CmdID`，不得修改既有基础帧格式或已发布 Payload 的字段顺序。
3. 上位机函数必须只封装已在下位机 `comm_protocol.c` 分发的命令。尚未实现的下位机能力只能保留为文档中的“预留”，不能混入正常任务路径。
4. 所有定点单位必须保持原样：位置 mm、姿态 cdeg、角速度 cdeg/s、速度 mm/s、时间 ms；禁止在传输层发送 float。
5. 每个新增命令必须同时补充命令常量、客户端函数、Payload 单元测试、实车 HIL 测试和本目录文档。

## 代码与测试

1. 模块命名按职责使用清晰前缀：通信放入 `protocol/`，视觉放入 `vision/`，编排放入 `app/`，业务模型放入 `domain/`。不要创建跨层的万能工具模块。
2. 高层 API 返回 `None` 表示 ACK 成功；查询 API 返回不可变数据对象。协议失败必须抛出既有异常，不能用静默默认值掩盖通信故障。
3. 串口请求由 `SerialTransport` 的互斥锁串行化，业务代码不得另起线程直接读写 `serial`。
4. 提交前必须执行 `conda run -n low_numpy python -m unittest discover -s tests -v`。实车联调由 `STM32_SERIAL_PORT` 环境变量显式启用。
