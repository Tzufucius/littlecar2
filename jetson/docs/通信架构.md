# 上下位机通信架构

## 链路关系

主控链路是 Jetson 或 Windows 上位机与 STM32F407 的 `USART6`。上位机 TX 接 STM32 `PC7 / USART6_RX`，上位机 RX 接 STM32 `PC6 / USART6_TX`，双方共地，使用 `115200 8N1`、无校验、无流控。不能将 RS-232 电平直接接入 STM32 的 3.3V UART。

```text
上位机业务（视觉、任务编排）
    -> protocol.client.Car
    -> protocol.transport.SerialTransport
    -> protocol.frame（二进制帧、CRC、流解析）
    -> USART6
    -> comm_pc（DMA + IDLE 原始字节桥接）
    -> comm_protocol（找帧、CRC、命令队列、ACK/DATA）
    -> advance_chassis / advance_motion / advance_world
    -> 电机与传感器驱动
```

上位机不会发送电机、舵机、OPS 或 WIT 的设备原始帧。它只发送统一的 `MSG_CMD`，STM32 侧在主循环的 `HostProtocol_Poll()` 中执行命令并由同一路 UART 回发。

## 应答关系

基础帧为 `5A A5 Version MsgType CmdSet CmdID Seq_L Seq_H Length Payload CRC_L CRC_H`。多字节字段使用小端，CRC 为 CRC16-Modbus，计算范围为 `Version` 到 `Payload` 末尾。

1. `SerialTransport` 分配递增 `Seq` 并发送 `MSG_CMD`。
2. STM32 对有效命令发送相同 `Seq` 的 `MSG_ACK`。
3. 查询命令在 ACK 后发送相同 `Seq + CmdSet + CmdID` 的 `MSG_DATA`。
4. 上位机以 `Seq`、命令组和命令 ID 三元组匹配应答；其他帧保留在接收队列中，避免粘包或状态上报被误丢弃。

当前 STM32 已实现 `SYSTEM`、`SAFETY` 和 `CHASSIS`。`SENSOR_GET_OPS`、`SENSOR_GET_IMU` 与周期 `MSG_STATUS` 尚未在 `comm_protocol.c` 中分发，不能作为当前联动流程的依赖。

## 上位机文件架构

```text
jetson/
  src/
    app/                 视觉和任务编排；只能调用高层 Car API
    domain/              业务事件模型；不依赖串口实现
    protocol/            与 STM32 comm_* 对应的上位机通信层
      commands.py        CmdSet、CmdID、ACK 和运动状态常量
      frame.py           帧编码、CRC、字节流解析、DATA 反序列化
      transport.py       串口读写、Seq、ACK/DATA 匹配与线程互斥
      client.py          System/Safety/Chassis 函数封装
      heartbeat.py       心跳后台服务
      types.py           对外数据对象
    vision/              相机和识别实现；不直接操作串口
  tests/                 离线单元测试与实车串口联调测试
  docs/                  架构与开发约束
```

命名与下位机保持同一职责划分：下位机 `comm_protocol.*` 对应上位机 `protocol/frame.py` 与 `protocol/transport.py`；下位机 `advance_*` 的公开能力对应上位机 `protocol/client.py` 的 `ChassisClient`。视觉、任务流程不得绕过 `Car` 直接拼包。

## 可调用能力

| 上位机函数 | STM32 命令 | 说明 |
| --- | --- | --- |
| `car.system.ping()` | `SYSTEM/0x01` | 验证帧、CRC、ACK 链路 |
| `car.system.heartbeat()` | `SYSTEM/0x02` | 刷新 STM32 300 ms 心跳看门狗 |
| `car.safety.estop()` / `safe_stop()` / `clear()` | `SAFETY/0x01~0x03` | 最高优先级的安全控制 |
| `car.chassis.enable()` / `stop()` | `CHASSIS/0x01~0x02` | 底盘使能与停车 |
| `set_motor_rpm()` / `move_mecanum()` | `CHASSIS/0x03~0x04` | RPM 调试控制 |
| `set_body_velocity()` / `set_world_velocity()` | `CHASSIS/0x05~0x06` | 物理速度控制 |
| `goto_pose()` / `cancel_goal()` | `CHASSIS/0x07~0x08` | 异步 world 目标点控制 |
| `reset_world_origin()` | `CHASSIS/0x09` | 传感器有效后建立 world 原点 |
| `get_motion_status()` | `CHASSIS/0x0B` | ACK 后读取 56 字节状态数据 |

`goto_pose()` 的 ACK 仅表示 STM32 接受了目标，必须继续调用 `get_motion_status()`，不能将 ACK 误判为已经到达。

## 推荐调用顺序

```python
from protocol import Car, WorldGoal

with Car(port="/dev/ttyTHS1") as car:
    car.system.ping()
    car.heartbeat.start()
    car.safety.clear()
    car.chassis.enable(True)
    car.chassis.reset_world_origin()
    car.chassis.goto_pose(WorldGoal(x_mm=500, y_mm=1000, use_yaw=False))

    # 任务循环中轮询 car.chassis.get_motion_status()，直到 ARRIVED 或异常状态。
    car.chassis.stop(mode=1)
```

`Car.close()` 会停止 `HeartbeatRunner` 并关闭串口。若调用方不使用后台心跳，必须自行以 50 至 100 ms 周期调用 `car.system.heartbeat()`；通信断开或 ACK 超时时必须停止继续下发运动命令。
